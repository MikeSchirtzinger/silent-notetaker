## Front-end specification (golden, do not change)

Matches `eval/js/melfeat.mjs` and `eval/dump_for_js.py` exactly.

| Property | Value |
|---|---|
| Sample rate | 16 kHz |
| Pre-emphasis | 0.97 |
| N_FFT | 512 |
| Win length | 400 |
| Hop length | 160 |
| Padding | reflect, PAD = 256 (numpy-style, edge sample not repeated) |
| Hann window | **periodic** (`0.5 - 0.5*cos(2πn/WIN)`, denominator = WIN) |
| Window placement | centred in N_FFT buffer (`left = (512-400)/2 = 56`) |
| Spectrum | power (`|X[k]|^2 = re^2 + im^2`) |
| Mel bands | **80**, slaney normalised (pre-baked from `mel_fb.json`) |
| Log guard | 2^-24 |
| Normalization | per-feature (per mel band) mean/std across time, +1e-5 stabiliser |
| Output dtype | f32 (computed in f64, cast at the end) |

### Why periodic Hann (not symmetric)

The symmetric Hann (`cos(2πn/(WIN-1))`) is what `nemotron-asr` uses and is what
`numpy.hanning` and `torch.hann_window(periodic=False)` produce. The TitaNet
NeMo training pipeline uses `librosa.filters.get_window('hann', 400, fftbins=True)`,
which is the **periodic** form (`fftbins=True`).

Using the symmetric form with TitaNet produces cosine ~0.9997 instead of 1.000000 —
a measurable mismatch that would degrade clustering.

**The two front-ends must never be unified.** A PR that "deduplicates" them
into one configurable frontend is a correctness bug.

## The "Two mel frontends, not one" table (from PRD)

| Property | TitaNet (this spike) | Nemotron (`nemotron-asr`) |
|---|---|---|
| Mel bands | **80**, slaney | **128**, slaney |
| Window | **periodic** Hann | **symmetric** Hann |
| Spectrum | magnitude → power (`S^2`) | power, log-guard 2^-24, no normalization |
| Normalization | **per-feature mean/std** | none |
| Filterbank | pre-baked `mel_fb.json` | computed from constants at runtime |

## Browser leg: validated 2026-06-04

The `WasmEmbedder` in `src/embedder_web.rs` follows the same pattern as
`nemotron-asr/src/backend_web.rs`:

- `TitaNetFrontend` (identical to native) runs in WASM and produces f32 features.
- `ort-web 0.2.1` bridges to onnxruntime-web 1.24 for async session creation and inference.
- The front-end produces f64 intermediate values and casts to f32 before inference,
  matching the native path exactly.

### Result

| Fixture | T | browser cosine vs ref | native cosine |
|---|---|---|---|
| `synthetic_spk1_sine440.json` | 401 | **1.00000012** | 1.000000 |
| `synthetic_spk2_sine880.json` | 401 | **0.99999988** | 1.000000 |
| `synthetic_spk3_noise.json` | 401 | **1.00000012** | 1.000000 |

**Worst browser cosine = 0.99999988** (`crossOriginIsolated === true`).

The sub-1e-6 deviation from exactly 1.0 is f32 rounding in the cosine reduction
(the `cosine_sim` accumulators are f32); the predicted tolerance was ≥ 0.9999 and
the measured result clears it by four orders of magnitude. ort-web's CPU kernel
produced bit-for-bit the same `embs` as the native `ort` CPU kernel, so the only
deviation visible at the embedding level is in the cosine accumulation, not in the
inference itself.

### How the COEP cross-origin problem was sidestepped

The harness serves **everything from one origin** (`http://127.0.0.1:8092`) with
`COOP: same-origin` + `COEP: credentialless`, which yields
`crossOriginIsolated === true` (enabling threaded WASM / SharedArrayBuffer for
`ort-wasm-simd-threaded.wasm`). The ort-web runtime assets — vendored by B3's
`vendor-ort-web.sh` (sha256-pinned) — are mounted at `/vendor/` on that same
origin, so COEP never blocks them. This is the path B3 recommended; it avoids the
separate-vendor-server CORS/COEP collision entirely.

`WasmEmbedder::create_with_dist(onnx, mel_fb, dist_base_url)` configures the
ort-web loader via `ort_web::Dist::new(dist_base_url)`. The base URL must be
**absolute** (the loader does `new URL(scriptName, baseUrl)`); the harness builds
it from `new URL("/vendor/", location.href)` so it stays same-origin.

### Egress proof (privacy boundary)

The browser Performance API recorded **15 resources, all from `127.0.0.1:8092`**:

- `cdn.pyke.io` (the ort-web default CDN): **0 fetches**
- `signal.pyke.io` (ort-web telemetry beacon): **0 fetches** (suppressed by the
  localhost guard in `_telemetry.js` and `ort::init().with_telemetry(false)`)

This matches the R5 privacy boundary: the embedder runs entirely local, no audio
/ embedding / mel tensor crosses a network boundary.

## Implementation notes

### Architecture

```
src/
  lib.rs              — crate root; cfg-gates native vs wasm modules
  error.rs            — typed Error enum (no anyhow in library code)
  audio.rs            — TitaNetFrontend (shared, platform-agnostic)
  embedder.rs         — native ort synchronous embedder
  embedder_web.rs     — wasm32 ort-web async WasmEmbedder
                        (create / create_with_dist for same-origin vendored assets)
  bin/
    validate.rs       — native CLI: reads ref/*.json, prints cosine table
harness/
  index.html          — browser-leg validation page (loads WasmEmbedder,
                        runs all fixtures, prints browser cosine to DOM + window.__B1)
  serve.py            — single-origin COOP/COEP server with virtual mounts
                        (/pkg, /vendor, /ref, /titanet.onnx, /mel_fb.json)
pkg/                  — wasm-pack --target web output (gitignored)
```

### Key implementation decisions

1. **f64 computation throughout the frontend.** The JS reference uses `Float64Array`;
   matching precision requires f64 in Rust until the f32 cast at the end. Using f32
   throughout produces a feature maxAbsDiff of ~1e-4 instead of 3.73e-9 — within
   the 1e-3 gate, but noisier. f64 is the right call for an exact port.

2. **realfft crate (same as nemotron-asr).** `RealFftPlanner::<f64>` matches the
   JS radix-2 FFT numerically. The key is the `f64` type parameter — the same
   crate used in `f32` mode would lose precision.

3. **Pre-baked filterbank from `mel_fb.json`, not computed.** The filterbank is
   computed by `librosa.filters.mel(..., htk=False, norm='slaney')` and baked into
   JSON. Recomputing it from scratch in Rust could introduce floating-point drift.
   Using the pre-baked matrix ensures the filterbank is bit-for-bit identical to
   the Python reference.

4. **No ndarray for the mel projection.** The mel basis is stored as
   `Vec<Vec<f64>>` and projected inline during the STFT loop, avoiding a separate
   power-spectrum allocation. This matches the JS reference loop structure.
