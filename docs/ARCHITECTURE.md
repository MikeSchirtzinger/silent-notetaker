# Architecture

This document describes how Silent Notetaker is built today on the `rust-refactor`
branch, where the hybrid Rust core is fully implemented. It supersedes the
pre-refactor version of this document.

The guiding principle, stated once so everything else follows from it:

> **Private by architecture, not by policy.** The cloud notetakers promise they
> won't misuse your audio. Silent Notetaker is built so the audio *physically
> cannot leave the device* — and so that anyone can verify that claim instead of
> trusting it.

Everything below is in service of making that claim **true** and **checkable**.

---

## 1. The trust boundary

The only thing that matters for the privacy claim is the **network egress
surface**: every origin the app can talk to. If that set contains no destination
your audio could flow to, the claim holds.

Here is the complete egress surface, derived from `registry/models.toml`
`network_origins` fields and the shipped `_headers`:

| Destination | Why | Carries your audio? |
|---|---|---|
| `cdn.jsdelivr.net` | Transformers.js runtime library | No |
| `unpkg.com` | Dexie IndexedDB wrapper | No |
| `cdn.pyke.io` | onnxruntime-web runtime loader | No |
| `huggingface.co` | Model weight downloads (initial fetch only; cached thereafter) | No |
| `*.hf.co` | Hugging Face CDN subdomains for model weights | No |
| `cdn-lfs.huggingface.co` | HF large-file storage CDN | No |
| `cdn-lfs-us-1.huggingface.co` | HF LFS CDN (US region) | No |
| `ws://localhost:8765` | Optional Claude bridge — a server **you** run, off by default | Only transcript text, only if you enable it, only to your own machine |

Audio is captured, turned into model inputs, and consumed in-process. It is
never serialized to a network call. That is the whole product.

**How this is enforced:** `_headers` is generated by `cargo xtask gen-headers`
from `registry/models.toml`. A hand-edited `_headers` that drifts from the
registry fails the CI `gen-headers --check` gate. The CSP `connect-src`
directive is enforced (not report-only) for the hosted build.

**The no-telemetry invariant.** There is no analytics origin in the registry,
no `fetch` to any metrics endpoint in the codebase, and no telemetry path in
any Rust policy crate. This is a structural invariant, not a runtime flag.

## 2. Hybrid architecture: Rust owns policy, JS executes

The refactor introduced a Rust workspace alongside the existing `index.html` UI,
implementing a strict division:

**Rust owns the law:** audio chunking policy, engine selection and device tiers,
streaming policy loops (including Voxtral's two-cap context-recycle), diarization
(SpeakerTracker, stop-time global recluster, rename/merge-by-rename), note trigger
extraction, smart-question scheduling, word-correction application, recording-session
state machine, storage schema and migrations, model registry, retry/backoff and
memory budgets, extension permissions, export formatting, and timestamp modes.

**JS keeps the hands:** DOM and the existing UI; `getUserMedia`, `getDisplayMedia`,
and AudioWorklet capture; the transformers.js host worker; ort-web loader glue; the
bridge WebSocket client; clipboard. These are permanent assignments by design, not
migration scaffolding. Each permanent JS module is listed in the workspace layout
so "permanent by design" is auditable.

The integration pattern is the strangler fig: Rust policies run under the unchanged
`index.html` UI. The UI calls wasm-bindgen surfaces exposed by `crates/silent-web`.
No user-visible behavior changed during the refactor; Appendix A of the PRD is the
parity contract.

### The silent-web boundary

`crates/silent-web` is the wasm-bindgen boundary crate. It exposes Rust policy
surfaces to the UI:

| Surface (in `silent_web.js`) | Backed by |
|---|---|
| `WasmSession` | `silent-core` session state machine |
| `WasmDiarization` | `silent-diarization` embedder + tracker |
| `WasmNotes` | `silent-notes` trigger extraction + open questions |
| `WasmQwen` | `silent-notes` smart-question scheduling + Qwen pipeline |
| `WasmStorage` | `silent-storage` IndexedDB reader/writer + migration |
| `WasmExports` | `silent-core` exporters + timestamp modes |
| `WasmHistory` | `silent-storage` meeting-history search |
| `WasmRegistry` | `silent-core` registry-driven engine selection |
| `WasmJsHostEngine` | `silent-inference` JS-host command dispatcher |
| `WasmDiag` | `silent-core` crash-diagnostics ring buffer + tracing |
| `WasmExtensionHost` | `silent-extension-sdk` + `silent-web` extension host |

**Glue files.** The following files in `apps/web/js/` are the permanent JS modules:

| File | Role |
|---|---|
| `capture.js` | Mic/tab capture, AudioWorklet bootstrap |
| `transformers-host.js` | transformers.js model host worker (the JS executor for Voxtral, Whisper, Moonshine, Qwen) |
| `ort-web-loader.js` | onnxruntime-web runtime loader for Rust/ort-web engines |
| `bridge-client.js` | Thin WebSocket client for `bridge.py` |

`apps/web/js/` also contains `session-engine.js` and `extension-host.js`, which are
thin glue wrappers that delegate all policy decisions to the corresponding wasm surfaces.

### Two mel frontends

`crates/silent-audio` houses two parameterized mel frontends that differ on nearly
every axis and must **never** be unified:

| Frontend | Used by | Bands | Window | Normalization |
|---|---|---|---|---|
| TitaNet 80-band | `silent-diarization` (speaker embedding) | 80 | Periodic Hann | Per-feature (slaney) |
| Nemotron 128-band | `crates/nemotron-asr` (ASR) | 128 | Symmetric Hann | Power spectrum (slaney) |

A PR that "deduplicates" these frontends is a correctness bug. They are
intentionally separate. See `docs/research/spike-titanet.md` for the full
validation evidence (cosine similarity 1.000000 vs the JS reference on `eval/`
fixtures).

## 4. Three model hosts

Every engine in the registry is assigned exactly one host:

| Host | What runs here | Selection rationale |
|---|---|---|
| `rust-ort-web` | Nemotron (streaming ASR), TitaNet (speaker embeddings) | Small INT8 models where CPU beats WebGPU (see `docs/research/spike-titanet.md` and nemotron-asr `reference/FINDINGS.md`); avoids GPU contention with Voxtral |
| `js-transformers` | Voxtral, Whisper ×4, Moonshine ×1, Qwen3 ×2 | transformers.js is the only runtime that runs these models in-browser; rewriting their runtime classes in Rust is explicitly out of scope |
| `js-sherpa` | SenseVoice (standalone and Dual-mode refiner) | sherpa-onnx Emscripten harness is the only practical path for SenseVoice in-browser |

The host assignment is recorded in `registry/models.toml` as a `host` field.
`silent-inference` reads this to dispatch commands to the correct executor.

A JS host adapter contains no policy: no thresholds, no retry logic, no
chunk-size or recycle decisions. Those arrive as typed commands and config from
Rust. Reviewers can verify this by reading `apps/web/js/transformers-host.js`.

## 5. Registry as single source of truth

`registry/models.toml` is the single source of truth for:

- Engine selection (which model loads for a given user choice)
- Device tier defaults (Qwen 0.6B on WASM-only / webgpu_low; 1.7B on webgpu_mid / webgpu_high)
- CSP generation (all `network_origins` fields aggregate into the `connect-src` allowlist)
- Egress manifest (what the app talks to and why)
- License display (per-model `license` field, `license_verified` flag)
- Cache verification policy (`cache-api` or `transformers-idb`; verify once per revision SHA)
- Performance budgets (per-model `perf_budget` table, gated in CI)
- Artifact hashes (per-file `sha256` and `size`, verified on first load)

**Adding a model within an existing family is a registry entry — zero code.** Adding
a new model family means implementing the engine trait in `silent-inference`; it does
not touch UI state management or audio capture.

### License verification status

All registry entries carry `license_verified = false` as of the initial population.
Task A2's verdict is recorded in comments on each entry. Mike must read the upstream
license and flip this flag; agents do not flip it. See `docs/research/model-licenses.md`
for the full A2 analysis.

## 6. Generated headers and CSP enforcement

`_headers` is generated by `cargo xtask gen-headers` from `registry/models.toml` and
must not be edited by hand. CI (`scripts/ci-local.sh` gate 10) fails if `_headers`
is stale relative to the registry.

**CSP status:** The base page CSP is enforced (not report-only), as of Phase 6 / Task
J3 (commit `71ac1dd`). The `--report-only` flag to `xtask gen-headers` retains a
rollback path.

The current enforced policy (from `_headers`):

```
Content-Security-Policy:
  default-src 'self';
  script-src  'self' 'unsafe-inline' 'wasm-unsafe-eval' blob:
              https://cdn.jsdelivr.net https://unpkg.com https://cdn.pyke.io;
  worker-src  'self' blob:;
  connect-src 'self' blob: data:
              https://cdn.jsdelivr.net https://unpkg.com https://cdn.pyke.io
              https://huggingface.co https://*.hf.co
              https://cdn-lfs.huggingface.co https://cdn-lfs-us-1.huggingface.co
              ws://localhost:8765;
  img-src     'self' data: blob:;
  media-src   'self' blob:;
  style-src   'self' 'unsafe-inline'
```

Notes:

- `connect-src` is the privacy linchpin. A locked `connect-src` means audio (and
  anything else) has nowhere unauthorized to go.
- `ws://localhost:8765` is kept in the hosted build. Decision log 2026-06-04:
  localhost is inside the user's trust boundary; omitting it silently breaks the
  bridge feature for hosted users.
- `'wasm-unsafe-eval'` is required by wasm-pack-compiled engines.
- `'unsafe-inline'` for scripts remains because `index.html` still contains
  inline script blocks. It will be removable once those blocks migrate to external
  modules.
- COEP is `credentialless` (not `require-corp`). Decision log 2026-06-04:
  `require-corp` breaks Hugging Face CDN redirects.

**Per-extension CSP.** Extensions run in null-origin sandboxed iframes. Their
per-extension network grants are applied as `<meta>` CSP inside the iframe context
via `GrantSet::connect_src`. See §7 for the known limitation.

## 7. Extension system

**Status: implemented (Phase 6 / Tasks J1–J3).**

Extensions run in null-origin sandboxed iframes (`sandbox="allow-scripts"` without
`allow-same-origin`). The host fetches the extension entrypoint source same-origin
and inlines it into the sandbox bootstrap (this avoids a COEP/CORS restriction on
`import()` from a null-origin context under `credentialless`).

The sandbox rests on three controls:

1. **Network denied by default.** Extensions cannot open a socket or `fetch` to any
   host until the user explicitly grants it. Grants are origin-scoped, visible at
   install time, and revocable from the extension manager UI.
2. **Isolated execution.** Extensions run in a sandboxed iframe with no direct DOM or
   global access. Communication with the host is exclusively `postMessage`. Raw audio,
   embeddings, mel tensors, and model activations are never sent across that boundary
   — they are not in the extension capability vocabulary at all.
3. **Declared capabilities.** `silent-extension-sdk` enforces the manifest; anything
   undeclared is denied at the boundary.

### Known limitation (J3 finding, pending follow-up)

A `srcdoc`/`blob:` iframe inherits the embedder's CSP in Chromium, and CSP
combines by intersection. A child context can only tighten, never widen, the base
page `connect-src`. This means per-extension network grants (which would relax
`connect-src` to an approved origin) **do not take effect** while extensions run in
`srcdoc` iframes: a granted origin is still blocked by the inherited base policy.

**The privacy-critical guarantee is met:** network deny-by-default is fully enforced;
the base page CSP is never relaxed. What is not yet working is the *relaxation* side
(grants). Making grants functional requires serving each extension from a distinct
same-origin document whose response header carries only that extension's CSP (no
inheritance). This is tracked as task J2b and is explicitly pending.

Grant persistence, consent UI, and the extension manager are fully implemented; the
CSP relaxation for granted origins awaits J2b.

See `docs/EXTENSIONS.md` for the full extension protocol specification.

## 8. Storage (silent-storage, schema v4)

`crates/silent-storage` reads and writes the `SilentNotetaker` IndexedDB using
`indexed_db_futures`. Storage bypasses Dexie entirely; the Dexie v2 schema is
handled only during migration.

**Schema v4 tables:**

| Table | Purpose |
|---|---|
| `meetings` | Meeting metadata (id, title, start/stop timestamps, speaker map) |
| `transcriptChunks` | Timestamped, speaker-attributed transcript segments |
| `notes` | Extracted decisions, actions, key points, open questions |
| `screenshots` | Binary screen capture blobs (normalized to `Uint8Array`) |
| `extensionGrants` | Per-extension network and capability grants (revocable) |

**Zero-loss Dexie v2 migration.** On first open of an existing Dexie v2 database,
`silent-storage` uses a proven two-phase reader (no IDB transaction held across a
non-IDB await) to migrate all four original tables. The UI offers an export-backup
before migration proceeds. The migration gate is enforced by the Phase 4 exit
criterion. See `docs/research/spike-storage.md` for the spike validation.

## 9. Verifiable builds (R8)

The single-file auditability story from the pre-refactor architecture is retired
honestly. `wasm-pack` ends the no-build-step era. It is replaced by something
stronger: **verify the binary you are running.**

**What this means:**

- `rust-toolchain.toml` pins the Rust toolchain, making wasm builds deterministic.
- CI publishes the sha256 of every deployed wasm artifact.
- The three-command reproduction path (fresh clone → matching hash) is documented
  in `CONTRIBUTING-RUST.md`.
- **In-app wasm hash display (settings/about panel): Phase K1 — pending.** The
  mechanism exists (CI publishes hashes); the UI surface that displays it has not
  yet been wired.

## 10. Distribution

Cloudflare Pages is the primary hosted distribution. The deploy gate is `cargo xtask
deploy-gate`:

- Fails if model weight files are present in the bundle (outside explicitly allowed
  tiny fixtures).
- Fails if any file exceeds Cloudflare Pages' 25 MB/file limit.
- Fails if `_headers` is stale relative to the registry.

Model artifacts are fetched from Hugging Face at runtime and cached in the browser.
The hosted bundle contains: app shell, WASM binaries, JS modules, CSS, `_headers`,
and registry metadata. No model weights.

## 11. What is NOT in scope

- **No hosted inference.** No backend transcription, no cloud LLM fallback, no
  server-side processing.
- **No telemetry.** The egress manifest (derived from the registry) is the proof.
- **No extension marketplace yet.** The sandbox, grant model, and reference extension
  are implemented. The hosted marketplace (discovery, signing, review) awaits Phase K5.
- **No Tauri native shell yet.** The roadmap includes a Rust shell for system-audio
  capture; it is not part of the current `rust-refactor` scope.
- **No pure-Rust model runtime for Voxtral.** Reimplementing transformers.js model
  classes in Rust is explicitly out of scope. Voxtral runs in `js-transformers` by
  design.
